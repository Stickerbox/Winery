# i18n Design — English / French Support

**Date:** 2026-03-24
**Status:** Approved

---

## Overview

Add English/French translation support to VinoVault. All UI-facing strings are extracted into typed translation files. A React context provides `t()` to client components; server components use a standalone helper that reads the same files. Language preference persists via a `lang` cookie. The Claude API call for wine image analysis is instructed to respond in the user's active language.

---

## Goals

- Every hardcoded UI string is replaced with a translation key
- Users can switch between English and French without a page reload
- Language preference persists across sessions (cookie-based, no URL changes)
- Claude API responses (wine name + tasting notes) are returned in the user's active language
- Zero new runtime dependencies

---

## Translation Files

### Structure

```
lib/i18n/
  en.ts        ← source of truth; defines the translations object shape
  fr.ts        ← typed as `typeof en`; TypeScript errors if a key is missing
  index.ts     ← exports Lang type, translations map, and getT() helper
```

### Typing Strategy

`en.ts` exports a plain object literal. `fr.ts` is typed `satisfies typeof en`, ensuring all keys are present and no extras are added. This gives autocomplete on every key path and a build-time error for missing translations.

### Key Namespaces

Keys are grouped by component/page to avoid collisions:

| Namespace | Coverage |
|---|---|
| `common` | App name, shared labels (e.g. "Cancel", "Delete") |
| `dashboard` | Header, sort options, search placeholder, welcome message |
| `wineForm` | Labels, placeholders, button states, error alerts |
| `wineModal` | Section headings ("Rating", "Tasting Notes"), action buttons |
| `wineGrid` | Empty state message |
| `login` | Card description, input placeholder, submit button |
| `share` | "Shared by", "Add to my wines", "View your collection", "Log in to add" |

## Language Context

### `LanguageProvider` (client component)

- Wraps the app in `app/layout.tsx`
- On mount, reads `lang` cookie; defaults to `"en"` if absent
- Exposes `{ lang, setLang, t }` via React context
- On `setLang(lang)`: updates state, writes `document.cookie = 'lang=<value>; path=/; max-age=31536000'`
- Re-renders all consuming components immediately — no page reload

### `useTranslations()` hook

- Returns `{ lang, setLang, t }` from context
- Used by all client components that need translations or the toggle

### Server component support

Server components (login page, share page) cannot use React context. They:

1. Call `cookies()` from `next/headers` to read the `lang` cookie
2. Pass the value to `getT(lang)` from `lib/i18n/index.ts`
3. Use the returned `t()` function for all string rendering

`getT()` is a pure function — it takes a language string and returns the corresponding translator. The same function is used internally by `LanguageProvider`, so client and server translations are always in sync.

## Claude API Integration

`analyzeWineImage` in `app/actions.ts`:

1. Reads the `lang` cookie from the current request via `cookies()` from `next/headers`
2. Appends a language instruction to the prompt:
   - English: `"Respond in English."`
   - French: `"Réponds en français."`
3. The wine `name` and `description` returned by Claude will be in the user's active language

This means wine tasting notes added while using French will be stored in French. This is intentional — the content reflects the language the user was using at the time of entry.

## Language Toggle Component

### `<LanguageToggle />`

A small client component rendered in three locations:

- `components/Dashboard.tsx` — below the `<WineGrid>`, centered
- `app/login/page.tsx` — below the login card
- `app/share/[token]/page.tsx` — below the wine card (requires converting to a client wrapper or passing lang as a prop from the server)

**Appearance:**

```
🌐  English  ·  Français
```

The active language label uses `text-zinc-700 dark:text-zinc-300` (readable). The inactive label uses `text-zinc-400 dark:text-zinc-500` (muted). Both are clickable. No border, no background — consistent with the app's minimal style.

## Constraints & Decisions

- **No URL routing.** Language is a user preference, not part of the URL. The share link `/share/<token>` works regardless of language.
- **No new dependencies.** The entire implementation uses React context, TypeScript, and Next.js built-ins.
- **Wine content is stored in the language it was entered.** Names and descriptions are not re-translated on language switch — they are user-entered (or AI-generated at time of entry) and treated as opaque data.
- **`lang` cookie is written client-side** on toggle. Server components always see the cookie from the previous request — a full page reload after toggle would pick it up. Since most of the app is client-rendered, this is not a problem in practice. The login and share pages will reflect the language on next load.
